## **BUG-001: Обязательное поле "likes" не соответствует спецификации**

| Поле | Значение |
|------|----------|
| **Серьёзность** | Critical (Критическая) |
| **Приоритет** | Highest (Наивысший) |
| **Компонент** | POST /api/1/item |
| **Статус** | OPEN |
| **Воспроизводимость** | 100% |

### Описание
API требует обязательное поле "likes" при создании объявления, что не соответствует предоставленной Postman-коллекции и ожидаемому поведению. Согласно коллекции, поле "statistics" является опциональным, а не обязательным.

### Шаги воспроизведения
1. Отправить POST запрос на `https://qa-internship.avito.com/api/1/item`
2. Тело запроса:
```json
{
  "sellerID": 123456,
  "name": "Test Ad",
  "price": 100
}
```
3. Проверить ответ сервера

### Фактический результат
```json
{
  "result": {
    "message": "поле likes обязательно",
    "messages": {}
  },
  "status": "400"
}
```
HTTP статус: **400 Bad Request**

### Ожидаемый результат
Создание объявления успешно, возвращается HTTP статус **200 OK** с ID объявления. Поле "likes" должно быть опциональным или иметь значение по умолчанию (0).

### Дополнительная информация
- Дефект затрагивает **100% позитивных тестов** на создание объявления
- В Postman-коллекции указан пример тела запроса без поля "likes"
- Аналогичная проблема наблюдается для полей **"viewCount"** и **"contacts"**
- Блокирует **60% всех тестов**, включая E2E и тесты производительности

### Затронутые тест-кейсы
TC-001, TC-002, TC-003, TC-004, TC-005, TC-006, TC-018, TC-021

## **BUG-002: Неправильный HTTP статус для несуществующих ресурсов**

| Поле | Значение |
|------|----------|
| **Серьёзность** | Major (Значительная) |
| **Приоритет** | High (Высокий) |
| **Компонент** | GET /api/1/item/:id, GET /api/1/statistic/:id, DELETE /api/2/item/:id |
| **Статус** | OPEN |
| **Воспроизводимость** | 100% |

### Описание
API возвращает 400 Bad Request для несуществующих ID вместо 404 Not Found, что нарушает REST-конвенции и HTTP-стандарты.

### Шаги воспроизведения
1. Отправить GET запрос на `/api/1/item/nonexistent-id-12345`
2. Отправить GET запрос на `/api/1/statistic/nonexistent-id-12345`
3. Отправить DELETE запрос на `/api/2/item/nonexistent-id-12345`

### Фактический результат
Во всех случаях: **HTTP статус 400 Bad Request**

### Ожидаемый результат
**HTTP статус 404 Not Found** с сообщением, что ресурс не найден:
```json
{
  "result": {
    "message": "Ad not found",
    "messages": {}
  },
  "status": "404"
}
```

### Затронутые тест-кейсы
TC-025, TC-041, TC-045

## **BUG-003: Несоответствие имен полей (sellerID vs sellerId)**

| Поле | Значение |
|------|----------|
| **Серьёзность** | Minor (Незначительная) |
| **Приоритет** | Low (Низкий) |
| **Компонент** | POST /api/1/item, GET /api/1/item/:id |
| **Статус** | OPEN |
| **Воспроизводимость** | 100% |

### Описание
В запросе на создание используется поле "sellerID" (верхний регистр D), а в ответе возвращается поле "sellerId" (нижний регистр d). Это нарушает консистентность API.

### Шаги воспроизведения
1. Отправить POST запрос с полем "sellerID"
2. Получить ответ и сравнить имена полей

### Фактический результат
- **Запрос:** `{"sellerID": 123456, ...}`
- **Ответ:** `{"sellerId": 123456, ...}`

### Ожидаемый результат
Единообразие именования полей (рекомендуется camelCase: "sellerId" в обеих частях)

### Дополнительная информация
Не блокирует функциональность, но может привести к путанице при разработке клиентов API и при автоматическом тестировании.

## **BUG-005: Отсутствует идемпотентность DELETE запроса**

| Поле | Значение |
|------|----------|
| **Серьёзность** | Major (Значительная) |
| **Приоритет** | High (Высокий) |
| **Компонент** | DELETE /api/2/item/:id |
| **Статус** | OPEN |
| **Воспроизводимость** | 100% |

### Описание
Согласно HTTP-спецификации (RFC 7231), DELETE метод должен быть идемпотентным. Повторное удаление одного и того же ресурса должно возвращать 404 Not Found, а не 400 Bad Request.

### Шаги воспроизведения
1. Создать объявление
2. Удалить его (первый DELETE) — 200 OK
3. Повторно отправить DELETE запрос с тем же ID

### Фактический результат
- **Первый DELETE:** 200 OK
- **Второй DELETE:** 400 Bad Request

### Ожидаемый результат
- **Первый DELETE:** 200 OK
- **Второй DELETE:** 404 Not Found

### Затронутые тест-кейсы
TC-048

## **BUG-006: Проблемы с параллельным созданием объявлений**

| Поле | Значение |
|------|----------|
| **Серьёзность** | Critical (Критическая) |
| **Приоритет** | High (Высокий) |
| **Компонент** | POST /api/1/item |
| **Статус** | OPEN |
| **Воспроизводимость** | 100% |

### Описание
При параллельном создании 10 объявлений (10 потоков) все запросы возвращают ошибку 400. Успешное создание возможно только при последовательных запросах. Это указывает на проблемы с конкурентной обработкой запросов.

### Шаги воспроизведения
1. Запустить 10 параллельных потоков
2. Каждый поток отправляет POST запрос на создание объявления
3. Проверить результаты

### Фактический результат
- **0 из 10 запросов** успешны
- Все возвращают 400 Bad Request с сообщением "поле likes обязательно"

### Ожидаемый результат
- **10 из 10 запросов** успешны
- Каждый возвращает 200 OK с уникальным ID

### Дополнительная информация
Проблема может быть связана с BUG-001 и конкурентным доступом к ресурсам. Даже после исправления BUG-001, API должен корректно обрабатывать конкурентные запросы.
